01What this document is about

The Nedbank AI demo was built with four isolated agent modules — Credit, Fraud, Social Sentiment, and Speech & CX — each living on its own tab, with its own chat thread, its own session ID, and its own backend endpoint. This is a perfectly valid architecture for a demo or internal prototype.

Production is a different contract. This document explains exactly where the isolated-module architecture breaks under production load and user expectations, what a routing-based architecture fixes, and what the migration path looks like.

TL;DR for business stakeholders: The demo has four specialist assistants and the user must decide which one to talk to. The production vision has one assistant that decides internally — the user just asks.

02The two architectures, side by side

🧩 Isolated Modules (Demo)
  • Four separate chat UIs, one per domain
  • User navigates to the right tab manually
  • Each tab has a fully independent conversation history
  • No shared memory between domains
  • Simple to build and reason about
  • Works well for a single-user demo
🔀 Routing Architecture (Production)
  • One chat UI for everything
  • A router agent decides which specialist to call
  • Shared memory: the router sees the full history
  • Cross-domain queries are possible natively
  • More complex to build, but scales linearly
  • Handles real multi-user concurrent load

03Production concerns — detailed breakdown

Concern Severity Isolated Modules Routing Architecture
Cross-domain queries HIGH Impossible without custom glue code per combination Native — router delegates to multiple agents and merges results
Shared memory / context HIGH Each tab has its own history; switching tabs loses context Single thread — every turn is visible to the router across all domains
Adding a new domain MEDIUM New tab + new page component + new backend router New specialist agent + one routing rule update
Session state at scale HIGH Four independent session IDs per user → 4× state management surface One session ID per user → clean horizontal scaling
Load balancer + sticky sessions HIGH LangGraph state is in-memory → requests must hit the same container (sticky sessions) Solved when checkpointing moves to Redis/PostgreSQL
Auth & multi-tenancy HIGH Mock auth only; everyone shares the same DB view Requires row-level security (RLS) in PostgreSQL per user role
Context window growth MEDIUM Per-tab histories grow independently; manageable but uncoordinated Single long thread can hit token limits — requires summarisation strategy
API key exposure HIGH Key in .env file on developer machines Must move to Vault / Azure Key Vault / AWS Secrets Manager
Cold start latency LOW 4 agent graphs compiled at startup — adds ~0.5s to container boot Same issue; use readiness probes to gate traffic until warm
Observability MEDIUM MLflow traces per domain are separate — hard to correlate a user's full session Single session trace contains all routing decisions and sub-agent calls

04The memory problem in detail

Memory is the single most impactful difference between the two architectures. Here's what happens to conversation state in each.

Isolated Modules — memory is siloed

# Session state today (per page, per tab)
credit_session_id    = "abc-111"   # isolated
fraud_session_id     = "abc-222"   # different thread, different memory
sentiment_session_id = "abc-333"   # different thread, different memory
speech_session_id    = "abc-444"   # different thread, different memory

# User asks Credit tab: "Is our approval rate declining?"
# → credit_agent answers. GREAT.
# User moves to Sentiment tab: "Is this related to social complaints?"
# → sentiment_agent has no idea credit was just discussed. Starts cold.
Real-world impact: An executive working a cross-domain analysis must mentally hold the context from one tab while switching to another, and then re-explain it to the second agent. The AI cannot connect the dots.

Routing Architecture — memory is unified

# Production: one session, one thread, managed by LangGraph + PostgreSQL checkpoint
user_session_id = "usr-007"

# Same conversation thread:
Turn 1: "Is our approval rate declining?"
         → router → credit_agent → "Yes, down 8% this quarter."

Turn 2: "Is this connected to the social complaints?"
         # Router sees Turn 1 in context
         → router → sentiment_agent → credit_agent
         → "Negative sentiment on credit topics rose 15% in the same period.
            The correlation is statistically significant."

Turn 3: "Which call centre team is handling these complaints?"
         # Still in same thread — router can cross-reference all three domains
         → router → speech_agent → "Agent Sipho's calls show the highest friction score."

What you need to move memory to production

LangGraph Checkpointing — swap SQLite for PostgreSQL
# Development (current)
from langgraph.checkpoint.sqlite import SqliteSaver
memory = SqliteSaver.from_conn_string(":memory:")

# Production — survives restarts, works across N containers
from langgraph.checkpoint.postgres import PostgresSaver
memory = PostgresSaver.from_conn_string(
    "postgresql://user:pass@pghost:5432/nedbank_ai"
)
graph = build_react_graph(...).compile(checkpointer=memory)

05How a routing agent works

A routing agent is a thin LLM layer that sits in front of all specialist agents. It reads the user's query, classifies intent, and delegates to one or more specialists.

The routing agent — skeleton implementation
# backend/app/agents/router_agent.py

from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage

ROUTER_PROMPT = """
You are an intent router for Nedbank's internal AI platform.
Given a user message, classify it into one or more domains:
- credit    → questions about loan/credit card applications
- fraud     → questions about suspicious transactions or fraud cases
- sentiment → questions about social media, customer opinion, brand perception
- speech    → questions about call centre recordings or CX scores
- general   → greetings, clarifications, meta questions

Respond ONLY with JSON: {"domains": ["credit"], "reason": "..."}
"""

async def route_query(query: str, history: list) -> list[str]:
    llm = ChatOpenAI(model="gpt-4o-mini")  # cheapest model — routing is simple
    response = await llm.ainvoke([
        SystemMessage(content=ROUTER_PROMPT),
        *history,
        HumanMessage(content=query)
    ])
    result = json.loads(response.content)
    return result["domains"]  # e.g. ["credit", "sentiment"]

# Then in the main handler:
async def unified_query(query: str, session_id: str):
    domains = await route_query(query, get_history(session_id))

    results = await asyncio.gather(*[
        invoke_specialist(domain, query, session_id)
        for domain in domains
    ])

    if len(results) == 1:
        return results[0]  # single domain — just return it
    else:
        return await synthesise(results, query)  # merge multiple answers
Key insight: The routing step is cheap. Using gpt-4o-mini for classification costs roughly 50× less than gpt-4o. The specialist agents that do the real work are still called with the full model. Total latency overhead for routing: ~200–400ms.

06Scaling under load — the sticky session problem

This is the most technically subtle production concern. The current demo runs as a single process. In production you run multiple containers behind a load balancer, and in-memory state breaks silently.

⚠ The problem — in-memory LangGraph state
# Current demo — LangGraph state lives in Python memory
# Works perfectly with 1 container. Breaks with 2.

Container A (handles Turn 1)   [history: ["Q1", "A1"]]
Container B (handles Turn 2)   [history: []]           ← empty! A1 is lost.

# User gets: "What were you asking about again?" from the AI
# or worse:  context from another user's session leaks in
✓ The fix — externalised checkpoint store
# Production — state stored in Redis or PostgreSQL, shared by all containers

Container A (handles Turn 1) → saves state to Redis["usr-007"]
Container B (handles Turn 2) → reads state from Redis["usr-007"]  ✓

# Any container can pick up any session. Stateless containers.
# This enables true horizontal scaling.

# Docker compose addition (production):
services:
  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    command: redis-server --maxmemory 512mb --maxmemory-policy allkeys-lru

07Auth & multi-tenancy

The demo uses mock authentication — a 900ms fake delay then immediate access. In a real banking system, two users must never see each other's data, and different roles must see different subsets.

08Context window management

Every message in the conversation history is sent to GPT-4o on every turn. A long session will eventually exceed the context window (128k tokens for GPT-4o), or simply become expensive and slow.

Summarisation strategy — keep relevant history, compress the rest
# A simple rolling summarisation approach
MAX_MESSAGES = 20  # keep last N raw turns

async def compress_history(messages: list) -> list:
    if len(messages) <= MAX_MESSAGES:
        return messages

    old = messages[:len(messages) - MAX_MESSAGES]
    recent = messages[-MAX_MESSAGES:]

    # Ask GPT to summarise the older part
    summary = await llm.ainvoke([
        SystemMessage("Summarise this conversation concisely for context."),
        *old
    ])

    # Replace old messages with a single summary message
    return [SystemMessage(f"[Earlier context] {summary.content}")] + recent

09Migration roadmap — from demo to production

Because the backend agents are already independent specialists, the migration does not require rewriting anything — only adding layers.

  • 1
    Externalise agent state
    Replace in-memory LangGraph state with PostgreSQL checkpointing. Deploy Redis for session cache. All existing agent logic is unchanged.
    Estimated effort: 1 week Unblocks: horizontal scaling
  • 2
    Build the routing agent
    A thin gpt-4o-mini classifier that maps queries to domain(s). Add a POST /api/query unified endpoint that delegates.
    Estimated effort: 1 week Unblocks: cross-domain queries
  • 3
    Replace mock auth with JWT
    Wire Azure AD / Auth0 to FastAPI. Add role claims to JWT payload. Add row-level security policies to PostgreSQL tables.
    Estimated effort: 2 weeks Unblocks: multi-user production deployment
  • 4
    Migrate frontend to unified UI
    Replace four-tab layout with a single chat interface. The backend routing handles domain selection transparently.
    Estimated effort: 1 week Unblocks: intuitive user experience
  • 5
    Move secrets to Vault / Azure Key Vault
    Remove API keys from .env. Inject at runtime via sidecar or init container.
    Estimated effort: 3 days Unblocks: security compliance
  • 6
    Add context summarisation
    Implement rolling window compression after N messages to control costs.
    Estimated effort: 2 days Unblocks: long-session stability

Verdict

The isolated-module architecture is the right choice for today's demo. It is explainable, visually clear, and easy to present to stakeholders. The routing architecture is the right choice for production — it scales linearly, enables compound queries, and matches the user experience pattern that people already understand from tools like ChatGPT.

The good news: the backend specialist agents require zero rewrite. The routing layer is purely additive. This is not a rebuild — it is a promotion.
