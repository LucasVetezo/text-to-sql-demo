01 — OverviewWhat this system is

NedCard Intelligent Suite is a production-ready, multi-agent AI platform that lets Nedbank analysts ask questions about credit applications, fraud transactions, social sentiment, and call-centre performance — in plain English — and receive intelligent, data-grounded answers in seconds.

The AI does not answer from training memory. It actively uses tools — querying a real database, transcribing audio, fetching social posts — then reasons about what it finds. This think → act → observe → repeat pattern is called a ReAct loop and is managed by LangGraph.

User: "Which branches have the worst decline rates, and is there a link to fraud?" │ ▼ Unified Router classifies intent → fans out to Credit + Fraud agents (parallel) │ ▼ LangGraph agents query the database → reason → produce answers │ ▼ GPT-4o synthesis merges both answers into one coherent response │ ▼ Frontend renders: domain badge → chart (above text) → animated answer
Two frontends, one backend. The React NedCard UI (port 3002) is the production-grade interface. The Streamlit app (port 8501) is the rapid-prototype demo interface. Both talk to the same FastAPI backend on port 8000.

06 — RoutersThe unified router (★ most important)

⚡
unified.py — POST /api/unified/query
backend/app/routers/unified.py

The entire intelligence of the routing layer. Every query from the React frontend (ChatPage) hits this one endpoint. Contains everything below.

Remove this → entire ChatPage stops working This is the production-path entry point

_DOMAINS — the domain registry

Ordered list. Fraud is checked before the credit wildcard so fraud queries don't fall through. Each entry has a key, label, compiled regex pattern, and reference to the compiled LangGraph graph.

_DOMAINS = [
  fraud     → re.compile(r"fraud|transact|suspicious|flagg|risk.?score|...")
  sentiment → re.compile(r"sentiment|social|twitter|x\.com|linkedin|post|...")
  speech    → re.compile(r"call.?cent|speech|transcript|cx|voice|recording|...")
  credit    → re.compile(r".*")   ← catches anything that didn't match above
]

_CONVERSATIONAL — the bypass

Regex matching greetings, thanks, farewells, meta-questions ("who are you"), and queries ≤3 words with no domain keywords. When matched: _conversational_reply() makes a direct OpenAI Chat Completions call (no LangGraph, no SQL), returns in ~50ms, sets agent_label = None → badge hidden.

_CREDIT_SIGNALS / _SENTIMENT_SIGNALS — cross-domain detection

Refined regex patterns used for cross-domain fan-out. A query matching both triggers parallel invocation of credit + sentiment agents followed by _synthesise(). Deliberately more specific than the broad domain patterns — prevents "what does credit mean to social perception" from triggering credit SQL tools unnecessarily.

_CHART_FOLLOWUP — context-aware routing

Detects follow-up references to already-rendered charts ("the bar chart", "what does that 48% mean", "according to the breakdown"). Routes to sentiment only — no credit agent invoked, because the user is reasoning about data already on screen.

_classify(query) — the routing decision function

Priority order (first match wins):
1. Conversational → general bypass (no specialist)
2. Chart follow-up + sentiment signal → sentiment only
3. Credit signals AND sentiment signals → [credit, sentiment] fan-out
4. First pattern match in _DOMAINS wins
5. If nothing matches → credit (the .*  wildcard)

_synthesise() — multi-agent answer merging

When ≥2 agents respond, GPT-4o is called once more with both answers and the original query. System prompt enforces: no forced "Executive Summary" header for simple questions, integrate numbers naturally, conversational tone for follow-ups, one "So what?" sentence only when it genuinely adds value.

agent_label — the badge signal

agent_label = " + ".join(d["label"] for d in domains if d.get("label"))
# "Credit Intelligence"                    ← single domain
# "Credit Intelligence + Social Sentiment" ← cross-domain
# None                                     ← conversational — badge hidden

ToolWhat it does

Layer 1: _validate_sql() — regex rejects INSERT/UPDATE/DELETE/DROP/ALTER before execution.

📊
chart_tools.py
backend/app/tools/chart_tools.py

Enhanced SQL tools that embed structured chart data in their return string using the string marker pattern. This is the mechanism that gets chart data from a LangGraph tool call all the way to the React frontend.

Remove this → charts stop rendering, but text answers still work

Why a string marker and not a ContextVar?

LangGraph's ToolNode uses asyncio.gather() internally, creating new async Tasks. Each Task gets a copy of Python's ContextVar context at creation — mutations inside those Tasks vanish when they finish. The string marker lives inside LangGraph's own message state (the messages list), which survives ainvoke() intact.

Tool return string format:

"Branches by loan amount: Sandton leads at R4.2M...

__CHART__:{"chart_type":"bar","title":"Loan Amount by Branch",
"x_key":"branch","y_key":"total_amount",
"rows":[{"branch":"Sandton","total_amount":4200000},...]}:__ENDCHART__"

After ainvoke(), extract_chart_from_messages() scans all ToolMessages
for the __CHART__ marker, parses the JSON, returns it as chart_data.

09 — RAGDocument & Audio Intelligence

The RAG (Retrieval-Augmented Generation) module lets users upload documents or audio recordings and ask questions against them — grounded on the actual file content, not model memory. Audio recordings (call centre, voice notes) are transcribed by Whisper-1 and analysed as call intelligence with a specialist 5-section assessor prompt.

Three-path submit logic in ChatPage.tsx:
PATH A — File attached: upload → index → auto-summary (or direct RAG if question typed alongside upload).
PATH B — Active document (banner shown): route question through RAG against the last uploaded doc.
PATH C — Normal unified agent query (no document context).
📄

09 — EvalsEvaluation framework

Three tiers of evaluation ensure the system produces correct, safe, and high-quality answers — not just responses that run without crashing.

TierWhenCostWhat it catches
Inline (Tier 1) Every query automatically Zero — no API calls Invalid SQL syntax, dangerous SQL keywords, empty/refusal answers
Batch (Tier 2) make eval on demand Small — 14 OpenAI calls Full-pipeline regressions across all 14 golden test cases, logged to MLflow with CSV artifact
LLM Judge (Tier 3) make eval-judge occasionally More tokens — GPT-4o judges each response Answer relevance and semantic similarity against ground truth — catches "valid but wrong" answers

PATH A — file attached: uploadDocument(file) → if question typed, skip auto-summary and call submitDocQuery() immediately (prevents duplicate responses); if no question, display auto-summary. Sets activeDocId/activeDocName amber banner for follow-up questions.

PATH B — activeDocId set: queryDocument(activeDocId, question) → RAG answer. Active doc banner stays until user clicks ×.

PATH C — no file, no active doc: queryAgent('/api/unified/query', ...) → normal multi-domain pipeline.

Typewriter streaming — punctuation-aware pacing

Backend returns the complete answer in one HTTP response. Client-side streaming reveals it character-by-character via recursive setTimeout with variable delays:

ConditionDelayEffect
Normal character18 ms~55 chars/sec — smooth reading pace
After . ! ?210 msSentence breath
After , ; :90 msClause beat
After \n130 msParagraph pause

Previous: setInterval at +6 chars / 16 ms = 375 chars/sec (mechanical). Current: recursive setTimeout where each character schedules its successor independently.

15 — SetupHow to start everything locally

Prerequisite: .env file must exist and contain a valid OPENAI_API_KEY. Copy .env.example and fill in your key.
Start order is critical — MLflow MUST start before the backend.
base_graph.py calls mlflow.set_experiment() at module import time. If MLflow is not already running when uvicorn imports the agents, the backend crashes with a connection error. Always follow the sequence below. 
# Step 1 — MLflow (port 5000) — MUST start FIRST
cd text_to_sql_demo
source .venv/bin/activate
.venv/bin/mlflow server --host 127.0.0.1 --port 5000 \
  --backend-store-uri sqlite:///mlflow.db --default-artifact-root ./mlruns &
# → http://localhost:5000

# Step 2 — Backend (FastAPI + LangGraph)
PYTHONPATH=backend .venv/bin/python -m uvicorn app.main:app --reload \
  --port 8000 --app-dir backend

# Step 3 — React frontend (production UI)
cd nedcard-ui
export PATH="/opt/homebrew/bin:$PATH"   # macOS only
npm run dev
# → http://localhost:3002

# Step 4 — Streamlit frontend (demo interface)
make dev-frontend
# → http://localhost:8501

# Optional — seed the database (first time or after schema changes)
make seed

# Optional — run evaluations (backend must be running)
make eval
Always use the venv. The project virtual environment (.venv/) has all required libraries. The system Python does not. Always use .venv/bin/python, .venv/bin/streamlit, .venv/bin/mlflow. Never use the bare python or pip without activating the venv first.
