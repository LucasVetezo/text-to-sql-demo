Isolated Modules vs Routing

Toggle between the two architectures to see how data flows and where the key differences lie.

The four-tab model. Each tab has its own session, its own history, and its own backend endpoint. Switching tabs breaks conversational context. The agents cannot share data or collaborate.

✓ Strengths

  • Simple to build and reason about
  • Easy to demo each capability independently
  • No routing complexity or latency
  • Failures are isolated — one agent crash doesn't affect others

✗ Limitations

  • Context is lost when switching between tabs
  • Cross-domain queries require manual effort from the user
  • 4× session state to manage under horizontal scaling
  • Adding a 5th domain means a 5th tab

The routing model. A single chat interface sends every query to a router agent. The router classifies intent and delegates to one or more specialist agents in parallel. All agents share the same session memory and database layer.

✓ Strengths

  • One session ID per user — clean horizontal scaling
  • Full conversation context visible across all domains
  • Cross-domain queries answered natively
  • Adding a domain = one new specialist + one routing rule

⚠ Requirements

  • PostgreSQL checkpoint store (not SQLite in-memory)
  • Redis for session cache under load
  • JWT auth + row-level security in DB
  • Context summarisation for long sessions
